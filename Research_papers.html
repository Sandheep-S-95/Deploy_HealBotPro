Why This Is Primarily an Information Retrieval System

The LLM (Groq / Llama) is used only twice in the entire pipeline: once for structured entity extraction (symptom parsing) and once for final report generation from pre-retrieved context. Every other step — graph traversal, dense vector indexing, metadata-filtered retrieval, document embedding, and patient history lookup — is classical and neural Information Retrieval. The pipeline performs 3 distinct retrieval passes per query before any generation occurs.

Pass 1 · Graph traversal (Neo4j Cypher) Pass 2A · Dense clinical retrieval (Pinecone) Pass 2B · Patient history retrieval (Pinecone) Dense vector indexing (SBERT all-MiniLM-L6-v2) Symptom→Disease knowledge graph Metadata-filtered top-k retrieval

I
Dense Vector Retrieval — The Core IR Backbone Foundational papers behind embedding-based passage retrieval in Pinecone
1
Core IR
Dense Passage Retrieval for Open-Domain Question Answering
Vladimir Karpukhin, Barlas Oğuz, Sewon Min, Patrick Lewis, Ledell Wu, Sergey Edunov, Danqi Chen, Wen-tau Yih
EMNLP 2020 — Proceedings of the Conference on Empirical Methods in Natural Language Processing, pp. 6769–6781
https://aclanthology.org/2020.emnlp-main.550/    arXiv: 2004.04906

This paper introduced Dense Passage Retrieval (DPR), proving that retrieval can be implemented using dense vector representations alone, abandoning sparse TF-IDF/BM25. The dual-encoder framework learns query and document embeddings such that inner-product similarity surfaces the most relevant passages. DPR outperformed BM25 by 9–19% absolute on top-20 retrieval accuracy.

Directly implemented in
embedding_model = SentenceTransformer('all-MiniLM-L6-v2') vector = embedding_model.encode(query_text).tolist() pinecone_index.query(vector=vector, top_k=5, include_metadata=True)
The Pinecone vector store is the production implementation of the DPR concept — every clinical context retrieval and patient history lookup in pass_2_pinecone_rag_retrieval() is a direct application of this paper's dense retrieval paradigm.
2
IR + Generation
Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks
Patrick Lewis, Ethan Perez, Aleksandra Piktus, Fabio Petroni, Vladimir Karpukhin, Naman Goyal, Heinrich Küttler, Mike Lewis, Wen-tau Yih, Tim Rocktäschel, Sebastian Riedel, Douwe Kiela
NeurIPS 2020 — Advances in Neural Information Processing Systems, Vol. 33, pp. 9459–9474
arXiv: 2005.11401    NeurIPS Proceedings PDF

This paper formalised the Retrieval-Augmented Generation (RAG) architecture — the exact paradigm HealBot Pro implements. It establishes that the retrieval step is primary and the generation is conditioned entirely on retrieved context. The RAG paper explicitly discusses medical knowledge indices as a target use case for this architecture.

Architectural blueprint for the entire pipeline
def run_consultation(user_text, user_id=None, doc_path=None): symptoms = extract_symptoms_with_llm(...) # query formulation diseases = pass_1_neo4j_graph_search(symptoms) # retrieval pass 1 clinical_ctx, patient_ctx = pass_2_pinecone_rag_retrieval(...) # retrieval passes 2a+2b report = generate_consultation_report(...context...) # generation conditioned on retrieval
The entire run_consultation() function is a direct implementation of the RAG pipeline described in this paper — retrieve first, generate second.
II
Sentence Embeddings for Semantic Indexing The paper behind the all-MiniLM-L6-v2 model used in the project
3
Core IR — Indexing
Sentence-BERT: Sentence Embeddings using Siamese BERT-Networks
Nils Reimers, Iryna Gurevych
EMNLP-IJCNLP 2019 — Proceedings, pp. 3982–3992, Hong Kong, China. Association for Computational Linguistics
https://aclanthology.org/D19-1410/    arXiv: 1908.10084

Sentence-BERT (SBERT) introduced siamese and triplet network structures on top of BERT to produce semantically meaningful, fixed-size sentence embeddings comparable via cosine similarity. This reduced semantic search from 65 GPU-hours to 5 seconds for 10,000-sentence corpora. The all-MiniLM-L6-v2 model used in HealBot Pro is a direct descendant of this architecture, trained using the sentence-transformers library produced by this paper's authors.

Directly instantiated in code
from sentence_transformers import SentenceTransformer embedding_model = SentenceTransformer('all-MiniLM-L6-v2') # Used for BOTH indexing and query encoding: vector = embedding_model.encode(query_text).tolist() # query side embedding = embedding_model.encode(text).tolist() # in upload_user_history()
Every vector stored in Pinecone and every query vector issued against it is produced by this paper's sentence transformer architecture. This is the indexing layer of the IR system.
III
Knowledge Graph Construction & Graph-Based Retrieval Papers behind Pass 1 — the Neo4j Symptom→Disease graph traversal
4
Core IR — Graph Retrieval
Building a Disease Knowledge Graph
Enayat Rajabi, Somayeh Kafaie
Studies in Health Technology and Informatics, Vol. 290, pp. 702–706, IOS Press, 2022. PMID: 37203473
https://pubmed.ncbi.nlm.nih.gov/37203473/    Full PDF (SemanticScholar)

This paper constructs a disease knowledge graph in Neo4j with nodes representing diseases, symptoms, drugs, and causes, with typed relationships such as "may cause" and "risk factor for." It demonstrates Cypher-based graph queries to answer complex clinical questions — the exact methodology deployed in HealBot Pro's first retrieval pass. The graph produced 45,000+ inferred relationships from structured medical databases.

Mapped to code — pass_1_neo4j_graph_search()
query = """ MATCH (s:Symptom)-[:INDICATES]->(d:Disease) WHERE s.name IN $symptoms RETURN d.name AS disease, collect(s.name) AS matched_symptoms, count(s) AS match_count ORDER BY match_count DESC LIMIT 3 """
The node types (Symptom, Disease), relationship type (INDICATES / INDICATES is analogous to "may cause"), and Cypher traversal pattern all follow the knowledge graph construction methodology described in this paper.
5
Core IR — Graph + Dense Hybrid
KG-RAG: Research on the Construction and Application of Retrieval-Enhanced Generation Model Based on Knowledge Graph
Zhao et al.
Scientific Reports (Nature Portfolio), Vol. 15, Article 39894, November 2025. DOI: 10.1038/s41598-025-21222-z
https://www.nature.com/articles/s41598-025-21222-z    PubMed: 41253865

This paper introduces a dual-channel retrieval mechanism combining (1) Dense Passage Retrieval for unstructured text and (2) graph-based semantic path retrieval from a knowledge graph — which is precisely the hybrid architecture of HealBot Pro. It validates this approach on medical QA benchmarks, achieving 81.3% accuracy on PubMedQA with a 6.8% improvement over vanilla RAG. The paper formalises the theoretical justification for performing both a graph pass and vector pass per query.

Validates HealBot's two-pass retrieval design
# Pass 1 — Graph channel (structured semantic paths) diseases = pass_1_neo4j_graph_search(symptoms) # Pass 2 — Dense channel (DPR-style vector retrieval) clinical_context, patient_history = pass_2_pinecone_rag_retrieval( symptoms, diseases, user_id )
HealBot Pro implements the exact "dual-channel retrieval" the paper advocates: structured graph traversal enriches and filters the dense retrieval, preventing irrelevant passage recall.

Medical Information Retrieval — Domain Validation Papers that establish the medical IR context and validate HealBot's architecture in healthcare
6
Medical IR Validation
Investigations on Using Evidence-Based GraphRAG Pipeline for Medical QA (USMLE)
Preprint — Multiple Authors
Preprint, 2024. Available via CloudFront / SSRN open access
Full PDF (open access)

This paper builds a medical GraphRAG system using Neo4j (with UMLS entities) and Pinecone for dense vector retrieval of medical textbook passages — an architecture that is nearly identical to HealBot Pro's. It uses the same two-component retrieval system: a graph database for structured entity traversal and a vector store for dense passage retrieval, then combines both contexts before LLM generation. The paper explicitly notes that its "IR system" outperforms LLM-only baselines in citation fidelity.

Parallel architecture to HealBot Pro
# Their system: Neo4j (UMLS entities) + Pinecone (textbook embeddings) # HealBot Pro: Neo4j (Symptom→Disease graph) + Pinecone (clinical context + patient history) # The same retrieval-before-generation principle: context_graph = graph_retrieval(query) # Neo4j Cypher context_dense = vector_retrieval(query) # Pinecone top-k report = llm_generate(context_graph + context_dense)
This paper serves as direct academic precedent for HealBot Pro's architecture in medical IR, confirming that Neo4j + Pinecone is a validated approach in peer-reviewed medical informatics research.
7
Survey — Medical RAG IR
Retrieval-Augmented Generation (RAG) in Healthcare: A Comprehensive Systematic Literature Review
Comprehensive Review (PRISMA, 30 peer-reviewed studies)
AI (MDPI Open Access), Vol. 6, Issue 9, Article 226, 2025. DOI: 10.3390/ai6090226
https://www.mdpi.com/2673-2688/6/9/226

A PRISMA-compliant systematic review of 30 peer-reviewed studies on RAG in clinical contexts. It categorises RAG architectures (naïve, advanced, modular) and identifies hybrid retrieval — combining knowledge graph integration with dense retrieval — as the most effective approach for medical QA. It specifically notes that graph-enhanced RAG architectures achieve higher diagnostic accuracy, citing MedRAG at 79.25% on differential diagnosis tasks.

Validates HealBot's design choices
# The survey classifies HealBot's architecture as "Modular RAG" with: # (1) Query processing — extract_symptoms_with_llm() # (2) Hybrid retrieval — Neo4j graph + Pinecone dense # (3) Multi-source integration — clinical DB + patient history # (4) Generation — generate_consultation_report() # This is the "most sophisticated architectural approach" per the survey.
Use this paper to cite that HealBot Pro implements state-of-the-art "Modular RAG" for healthcare — a pattern explicitly validated and recommended by a formal systematic review.
